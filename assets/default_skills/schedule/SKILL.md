# Schedule — 工作计划管理技能

> 核心理念: SCHEDULE.md 是你的工作记忆。开工前必读，做完必更新，发现问题必插入。
> 防止长对话中遗忘任务，确保问题按优先级逐个击破。

## 核心工作流

### 1. 开工前: 加载回顾 (50% 概率触发)

**每次用户给你新任务或开始新工作前**，以约 50% 的概率调用 `schedule_load` 回顾计划。以下情况必定触发（不受概率限制）：
- 用户明确提到"计划/进度/任务/schedule"
- 用户给出一个全新的多步需求
- 上次回顾距今已超过 5 轮对话

回顾后简要汇报：
- 当前进度：待办数量、最高优先级任务
- 确认本次聚焦目标
- 如果用户的新需求不在计划中，评估优先级后插入

### 2. 规划阶段: 问题分类

收到大型需求时（多个子任务、一整块功能），先做问题拆解：
1. 将需求拆分成独立的子任务
2. 为每个子任务评估优先级 (P0/P1/P2)
3. 调用 `schedule_add_task` 逐条插入
4. 向用户确认优先级和顺序
5. 然后从最高优先级开始逐个击破

### 3. 执行阶段: 逐个击破

**做一件事之前**：确认它对应 SCHEDULE 中的哪个任务。
**做完一件事之后**：立即 `schedule_complete` 标记完成。

### 4. 中途发现新问题: 即时插入

工作过程中发现的 Bug、遗漏、优化点：
1. 立即评估优先级
2. 调用 `schedule_add_task` 插入（不要靠记忆，会忘）
3. 简短告知用户："发现了 xxx，已加入计划，优先级 P1"
4. 继续当前任务（除非新问题是 P0 阻塞性的）

### 5. 防遗忘检查

每完成 3 次工具调用后，内心核实：
- 当前是否偏离了计划中的目标？
- 是否有更高优先级的任务被搁置？
- 如有偏离，主动提醒用户

### 6. 阶段性汇报

以下情况触发 `schedule_review`：
- 用户问进度
- 一个 P0 任务完成
- 连续工作超过 5 轮无汇报

## 优先级规则

| 级别 | 标记 | 含义 | 处理原则 |
|------|------|------|----------|
| P0 | 🔴 | 阻塞性/紧急 | 立即处理，不可被打断 |
| P1 | 🟡 | 重要但不紧急 | 当前 P0 完成后按序执行 |
| P2 | 🟢 | 一般/可延后 | 有空再做，不主动提起 |
| Done | ✅ | 已完成 | 归档到底部，保留记录 |

## SCHEDULE.md 格式规范

```markdown
# 工作计划
> 最后更新: 2026-06-09 20:45

## 🔴 P0 - 紧急
- [ ] 任务描述 `#标签` — 备注
- [ ] 另一个任务

## 🟡 P1 - 重要
- [ ] 任务三 `#feature`

## 🟢 P2 - 一般
- [ ] 任务四

## ✅ 已完成
- [x] 完成的任务 — 2026-06-09 完成
```

## 边界

- 不要在 SCHEDULE.md 中记录对话内容或技术细节，只记录任务名+简要备注
- 优先级由用户最终决定，模型仅建议
- 单个 SCHEDULE.md 最多保留 50 条已完成记录，超出时自动归档
- 不修改用户自建的 schedule 文件的格式风格，只在其框架内操作
